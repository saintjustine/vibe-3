# PRD - 공공직군 행정업무 슈퍼앱

## 1. 문서 정보

| 항목 | 내용 |
| --- | --- |
| 문서 목적 | 공공직군 행정업무 슈퍼앱의 기능 요구사항 정의 |
| 대상 버전 | MVP 0.1 |

## 2. 제품 개요

공공직군 행정업무 슈퍼앱은 팀 단위 행정업무에서 반복적으로 발생하는 일정 공유, 엑셀 파일 정리, 민원 응대 문안 작성, 공공행정 뉴스 수집을 하나의 웹 애플리케이션에서 처리하는 내부 업무 도구이다.

핵심 목표는 다음과 같다.

| 목표 | 설명 |
| --- | --- |
| 업무 가시성 향상 | 팀원 휴가, 근무, 출장 일정을 캘린더에서 한눈에 확인한다. |
| 반복 업무 자동화 | 엑셀 분할, 병합 작업을 수동 작업 없이 처리한다. |
| 민원 대응 품질 표준화 | 민원 매뉴얼 기반으로 답변 방향과 응대 스크립트를 작성한다. |
| 정보 수집 자동화 | 공공행정 관련 뉴스를 매일 아침 자동 수집한다. |

## 3. 사용자와 권한

| 사용자 | 주요 목적 | 권한 |
| --- | --- | --- |
| 팀원 | 본인 일정 등록, 팀 일정 확인, 엑셀 자동화 사용, 민원 응대 초안 생성 | 본인 일정 생성/수정, 공용 기능 사용 |
| 팀장 | 팀 일정 확인, 인력 공백 확인, 업무 현황 점검 | 전체 일정 열람, 필요 시 일정 조정 |
| 관리자 | 사용자, 매뉴얼, 수집 설정, 시스템 운영 관리 | 전체 데이터 관리, 설정 변경 |

MVP에서는 단순 역할 기반 권한을 사용한다. 외부 SSO, 전자결재 연동, 기관 인증 연동은 후속 범위로 둔다.

## 4. MVP 범위

### 4.1 포함 범위

| 영역 | MVP 포함 기능 |
| --- | --- |
| 팀원 스케줄 관리 | 캘린더 조회, 일정 등록/수정/삭제, 일정 유형 구분, 팀원별 필터 |
| 엑셀 업무 자동화 | 엑셀 업로드, 특정 컬럼 기준 분할, 여러 엑셀 파일 병합, 결과 다운로드 |
| 민원 대응 챗봇 | 민원 매뉴얼 업로드 또는 등록, 질문 입력, 대응 방향 및 응대 스크립트 생성 |
| 뉴스 기사 수집 | 공공행정 키워드 기반 기사 수집, 중복 제거, 아침 수집 작업, 목록 조회 |
| 공통 | SQLite 저장, API 서버, React 기반 웹 화면, 기본 운영 문서 |

### 4.2 제외 범위

| 제외 항목 | 제외 사유 |
| --- | --- |
| 기관 SSO 연동 | 초기 MVP 검증 이후 보안 정책 확인 필요 |
| 전자결재 연동 | 기관별 시스템 차이가 커 별도 분석 필요 |
| 모바일 앱 | 우선 반응형 웹으로 대응 |
| 대규모 다중 기관 운영 | SQLite 기반 MVP 이후 PostgreSQL 등으로 확장 검토 |
| 실시간 푸시 알림 | MVP에서는 화면 조회와 수동 새로고침 중심 |

| ID | 요구사항 | 우선순위 |
| --- | --- | --- |

| SCH-002 | 사용자는 본인의 휴가, 근무, 출장, 교육, 기타 일정을 등록할 수 있어야 한다. | Must |
| SCH-003 | 사용자는 본인이 등록한 일정을 수정하거나 삭제할 수 있어야 한다. | Must |
| SCH-004 | 팀장과 관리자는 팀 전체 일정을 조회할 수 있어야 한다. | Must |
| SCH-005 | 일정은 제목, 유형, 시작일시, 종료일시, 작성자, 메모를 포함해야 한다. | Must |
| SCH-006 | 사용자는 팀원, 일정 유형, 기간 기준으로 일정을 필터링할 수 있어야 한다. | Should |
| SCH-007 | 동일 사용자의 중복 일정이 있으면 경고를 표시해야 한다. | Should |
| SCH-008 | 캘린더는 월간 보기와 목록 보기를 제공해야 한다. | Should |
| SCH-009 | 일정 변경 이력은 운영 로그로 남길 수 있어야 한다. | Could |

### 주요 화면

| 일정 등록 모달 | 일정 유형, 기간, 제목, 메모 입력 |

### 수용 기준

| 조건 | 기대 결과 |
| --- | --- |

| 팀원 필터를 선택한다. | 선택된 팀원의 일정만 표시된다. |
| 종료일시가 시작일시보다 빠르다. | 저장되지 않고 검증 오류가 표시된다. |

## 5.2 엑셀 업무 자동화

### 기능 설명

엑셀 파일을 업로드한 뒤 특정 컬럼 기준으로 파일을 나누거나, 여러 엑셀 파일을 하나로 합치는 기능이다.

### 요구사항

| ID | 요구사항 | 우선순위 |
| --- | --- | --- |
| XLS-001 | 사용자는 `.xlsx` 파일을 업로드할 수 있어야 한다. | Must |
| XLS-002 | 업로드된 엑셀의 시트명과 컬럼명을 미리 확인할 수 있어야 한다. | Must |
| XLS-003 | 사용자는 특정 컬럼을 선택해 값별로 엑셀 파일을 분할할 수 있어야 한다. | Must |
| XLS-004 | 분할 결과는 ZIP 파일로 다운로드할 수 있어야 한다. | Must |
| XLS-005 | 사용자는 여러 엑셀 파일을 업로드해 하나의 파일로 병합할 수 있어야 한다. | Must |
| XLS-006 | 병합 시 컬럼 구조가 다른 파일은 경고 처리해야 한다. | Should |
| XLS-007 | 처리 결과는 파일명, 행 수, 처리 시간, 성공/실패 여부를 기록해야 한다. | Should |
| XLS-008 | 개인정보가 포함된 파일은 처리 후 임시 저장소에서 삭제되어야 한다. | Must |

### 처리 규칙

| 작업 | 규칙 |
| --- | --- |
| 분할 | 선택한 컬럼의 고유값 단위로 파일을 생성한다. |
| 병합 | 첫 번째 파일의 컬럼 구조를 기준으로 병합한다. |
| 파일명 | 금지 문자를 제거하고 안전한 파일명으로 변환한다. |
| 임시 파일 | 다운로드 완료 또는 일정 시간이 지난 뒤 삭제한다. |

### 수용 기준

| 조건 | 기대 결과 |
| --- | --- |
| `부서` 컬럼 기준으로 분할한다. | 부서별 `.xlsx` 파일이 ZIP으로 생성된다. |
| 컬럼 구조가 다른 파일을 병합한다. | 병합 전 경고가 표시되고 사용자가 중단할 수 있다. |
| 빈 파일을 업로드한다. | 처리하지 않고 오류 메시지를 반환한다. |

## 5.3 민원 대응 챗봇

### 기능 설명

첨부 또는 등록된 민원 매뉴얼을 기준으로 민원 상황에 대한 대응 방향, 확인 사항, 응대 스크립트 초안을 제공하는 기능이다.

### 전제

현재 프로젝트에는 민원 매뉴얼 파일이 첨부되어 있지 않다. MVP에서는 매뉴얼을 업로드하거나 텍스트로 등록할 수 있는 구조를 전제로 한다.

### 요구사항

| ID | 요구사항 | 우선순위 |
| --- | --- | --- |
| BOT-001 | 관리자는 민원 매뉴얼 문서를 등록할 수 있어야 한다. | Must |
| BOT-002 | 사용자는 민원 내용을 입력하고 대응 방향을 요청할 수 있어야 한다. | Must |
| BOT-003 | 답변은 매뉴얼 근거, 대응 절차, 응대 스크립트로 구분되어야 한다. | Must |
| BOT-004 | 답변에는 참조한 매뉴얼 섹션 또는 문서명이 표시되어야 한다. | Must |
| BOT-005 | 근거가 부족한 질문은 임의 답변 대신 추가 확인 필요로 표시해야 한다. | Must |
| BOT-006 | 사용자는 전화 응대, 이메일 답변, 내부 보고용 문체를 선택할 수 있어야 한다. | Should |
| BOT-007 | 챗봇 질문과 답변 이력은 개인정보를 제외하고 저장할 수 있어야 한다. | Should |
| BOT-008 | 민원인의 주민번호, 연락처 등 민감정보 입력 시 마스킹 안내를 제공해야 한다. | Must |

### 답변 형식

| 섹션 | 내용 |
| --- | --- |
| 상황 요약 | 사용자가 입력한 민원 내용을 업무 관점에서 요약 |
| 확인 사항 | 추가로 확인해야 할 자료, 법령, 담당 부서 |
| 대응 방향 | 매뉴얼 근거에 따른 처리 방향 |
| 응대 스크립트 | 실제 말하기 또는 회신에 사용할 수 있는 문안 |
| 근거 | 참조한 매뉴얼 문서명, 섹션, 키워드 |

### 수용 기준

| 조건 | 기대 결과 |
| --- | --- |
| 매뉴얼에 있는 민원 유형을 질문한다. | 근거와 함께 대응 스크립트가 생성된다. |
| 매뉴얼에 없는 내용을 질문한다. | 확인 필요 답변을 반환하고 임의 절차를 만들지 않는다. |
| 개인정보가 포함된 질문을 입력한다. | 민감정보 마스킹 안내를 표시한다. |

## 5.4 뉴스 기사 수집

### 기능 설명

공공행정 관련 뉴스를 매일 아침 자동으로 수집하고, 중복 제거 후 목록으로 제공하는 기능이다.

### 요구사항

| ID | 요구사항 | 우선순위 |
| --- | --- | --- |
| NEWS-001 | 관리자는 수집 키워드를 등록하고 수정할 수 있어야 한다. | Must |
| NEWS-002 | 시스템은 매일 아침 지정된 시간에 뉴스를 수집해야 한다. | Must |
| NEWS-003 | 수집 대상은 공공행정, 지방행정, 정책, 민원, 디지털정부 관련 키워드를 포함해야 한다. | Must |
| NEWS-004 | 동일 URL 또는 동일 제목 기사는 중복 저장하지 않아야 한다. | Must |
| NEWS-005 | 사용자는 수집된 기사 목록을 날짜, 키워드, 출처 기준으로 조회할 수 있어야 한다. | Must |
| NEWS-006 | 기사 제목, 출처, 발행일, URL, 요약, 수집 키워드를 저장해야 한다. | Should |
| NEWS-007 | 수집 실패 시 실패 사유와 시간을 로그로 남겨야 한다. | Must |
| NEWS-008 | 언론사 약관과 저작권을 고려해 본문 전문 저장은 기본 비활성화해야 한다. | Must |

### 수집 정책

| 항목 | 정책 |
| --- | --- |
| 기본 수집 시간 | 매일 08:00 |
| 기본 키워드 | 공공행정, 지방행정, 민원, 행정안전부, 디지털정부, 공공서비스 |
| 중복 기준 | URL 우선, URL이 없으면 제목과 출처 조합 |
| 저장 범위 | 제목, 링크, 발행일, 출처, 짧은 요약, 키워드 |

### 수용 기준

| 조건 | 기대 결과 |
| --- | --- |
| 매일 08:00이 된다. | 등록된 키워드 기준으로 기사 수집 작업이 실행된다. |
| 이미 저장된 URL이 다시 수집된다. | 새 기사로 중복 저장하지 않는다. |
| 수집 API가 실패한다. | 오류 로그를 저장하고 화면에서 확인할 수 있다. |

| 민원 매뉴얼 | 관리자 삭제 전까지 저장 |
| 뉴스 기사 메타데이터 | 운영 DB에 지속 저장 |
| 시스템 로그 | 기간 정책에 따라 순환 삭제 |
